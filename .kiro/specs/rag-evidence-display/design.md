# RAG证据来源展示功能 - 技术设计文档

## 1. 功能概述

当用户询问知识性问题（如"访问关系如何进行开通提单"）时，系统会从RAG知识库中检索相关文档。本功能旨在：
1. 在API响应中返回RAG检索到的证据来源信息
2. 在前端页面展示这些证据来源（文档标题、相关度评分等）
3. 支持用户点击证据来源，预览完整的文档内容

## 4. 实现细节

### 4.1 文档ID生成策略

**方案1: 基于文件路径的哈希**
```python
import hashlib

def generate_doc_id(file_path: str) -> str:
    """生成文档ID（基于文件路径的SHA256哈希）"""
    return hashlib.sha256(file_path.encode()).hexdigest()[:16]
```

**方案2: 使用向量数据库的ID**
```python
def generate_doc_id(chroma_id: str) -> str:
    """直接使用ChromaDB的文档ID"""
    return chroma_id
```

**推荐**: 方案2（使用ChromaDB的ID），因为：
- 避免重复计算
- 与向量数据库保持一致
- 查询效率更高

## 5. 性能考虑

### 5.1 文档预览优化
- **预览文本长度**: 限制为200字符
- **文档内容缓存**: 使用LRU缓存，最多缓存100个文档
- **懒加载**: 只在用户点击时才加载完整文档

### 5.2 SSE事件优化
- **批量发送**: 将多个证据来源合并为一个事件
- **压缩**: 对大文档内容进行gzip压缩
- **限流**: 限制证据来源数量（最多5个）

### 5.3 前端渲染优化
- **虚拟滚动**: 如果证据来源很多，使用虚拟滚动
- **防抖**: 搜索和过滤操作使用防抖
- **懒渲染**: 证据面板默认折叠，点击展开

## 6. 安全考虑

### 6.1 文档访问控制
- 验证文档ID的合法性
- 防止路径遍历攻击
- 限制文档大小（最大10MB）

### 6.2 XSS防护

## 10. 正确性属性

*属性是系统在所有有效执行中应保持为真的特征或行为——本质上是关于系统应该做什么的形式化陈述。属性作为人类可读规范和机器可验证正确性保证之间的桥梁。*

### 属性 1: RAG检索返回有效证据来源

*对于任意*知识性问题，当RAG引擎检索文档时，系统应返回有效的证据来源列表，其中每个证据来源包含唯一的文档ID、文件名、有效的相关度评分（0-1范围）和预览文本（不超过200字符）。

**验证需求**: 1.1, 1.2, 1.3, 2.1, 2.2, 2.3, 2.4

### 属性 2: 低相关度文档过滤

*对于任意*RAG检索结果，系统应过滤掉相关度评分低于0.05的文档，确保返回的证据来源列表中所有文档的相关度评分都大于等于0.05。

**验证需求**: 1.4

### 属性 3: 证据来源数量限制

*对于任意*RAG检索结果，无论检索到多少文档，系统返回的证据来源数量应不超过5个。

**验证需求**: 1.5

### 属性 4: 元数据完整性

*对于任意*包含元数据的文档，其对应的证据来源对象应包含元数据字段（来源路径、创建时间、文件大小）。

**验证需求**: 2.5

### 属性 5: SSE事件流正确性

*对于任意*完成RAG检索的请求，系统应通过SSE流发送evidence_sources事件，该事件包含type字段（值为"evidence_sources"）和sources数组（包含所有证据来源对象），且该事件应在content事件之前发送。

**验证需求**: 3.1, 3.2, 3.3, 3.4

### 属性 6: 无文档时不发送证据事件

*对于任意*未检索到相关文档的请求，系统应不发送evidence_sources事件。

**验证需求**: 3.5

### 属性 7: 文档查询API正确性

*对于任意*有效的文档ID，文档查询API应返回包含文档ID、文件名、完整内容和元数据的响应；对于任意无效的文档ID，API应返回404错误和描述性错误消息。

**验证需求**: 4.2, 4.3, 4.4, 12.4

### 属性 8: 前端证据面板渲染

*对于任意*接收到的evidence_sources事件，前端应渲染证据来源面板，显示证据来源数量，并为每个证据来源渲染包含文件名、相关度评分和预览文本的卡片。

**验证需求**: 5.1, 5.2, 5.3, 5.4

### 属性 9: 文档预览交互

*对于任意*证据卡片的点击操作，前端应发送GET请求到/api/v1/knowledge/document/{doc_id}，加载成功后在模态框中显示文档标题和完整内容。

**验证需求**: 6.1, 6.3

### 属性 10: 文档ID一致性

*对于任意*文档，系统应使用ChromaDB的文档ID作为证据来源的文档ID，且同一文档在多次检索中应返回相同的文档ID，不同文档应有不同的文档ID。

**验证需求**: 7.1, 7.2, 7.3, 7.4

### 属性 11: LRU缓存行为

*对于任意*文档查询序列，文档服务应使用LRU缓存策略，缓存最多100个文档，已缓存的文档从缓存返回，未缓存的文档从数据源加载并缓存，当缓存满时移除最久未使用的文档。

**验证需求**: 8.1, 8.2, 8.3, 8.4, 8.5

### 属性 12: 路径遍历攻击防护

*对于任意*包含路径遍历攻击模式（如../、..\\）的文档ID请求，系统应验证文档ID的合法性并拒绝请求，返回错误响应。

**验证需求**: 9.1, 9.2

### 属性 13: HTML内容转义

*对于任意*包含HTML标签的文档内容，前端应对内容进行HTML转义，确保标签不被执行为脚本。

**验证需求**: 9.3

### 属性 14: RAG检索容错性

*对于任意*RAG检索失败的情况，系统应记录错误日志并继续处理用户请求，生成回答，不因证据来源功能而中断服务。

**验证需求**: 10.5, 12.3

### 属性 15: 批量事件发送

*对于任意*包含多个证据来源的RAG检索结果，系统应将所有证据来源合并为一个evidence_sources事件批量发送，而不是发送多个事件。

**验证需求**: 11.2

### 属性 16: 懒加载行为

*对于任意*证据卡片，前端应仅在用户点击该卡片时才发送文档查询请求加载完整文档，而不是在渲染证据面板时预加载所有文档。

**验证需求**: 11.4

### 属性 17: 消息持久化完整性

*对于任意*包含证据来源的聊天消息，系统在保存消息时应在消息对象中包含evidenceSources字段，存储所有关联的证据来源；加载历史消息时应渲染证据来源面板，且证据卡片可点击查看文档。

**验证需求**: 13.1, 13.2, 13.3, 13.4

### 属性 18: 相关度评分可视化

*对于任意*证据卡片，前端应以可视化方式显示相关度评分，并根据评分范围使用不同颜色标识：高于0.7使用高亮颜色，0.4-0.7使用中性颜色，低于0.4使用低亮度颜色。

**验证需求**: 14.1, 14.2, 14.3, 14.4

### 属性 19: 证据来源排序

*对于任意*证据来源列表，前端应默认按相关度评分从高到低排序显示。

**验证需求**: 14.5

### 属性 20: 监控日志记录

*对于任意*RAG检索操作，系统应记录检索耗时和返回的证据来源数量到日志；对于任意文档查询操作，系统应记录查询耗时到日志；对于任意证据卡片点击和文档预览加载，系统应记录点击次数和加载时间到指标系统。

**验证需求**: 15.1, 15.2, 15.3, 15.4, 15.5
