# 默认聊天窗口快捷问题设计

## 背景

当前默认聊天窗口已具备欢迎消息、输入框、会话恢复和新对话能力，但新用户进入页面后缺少明确的首屏引导。用户常见诉求集中在网络故障诊断、访问关系查询、访问开通提单三个方向，且问题描述通常存在固定模板。

本设计在不改动后端接口的前提下，为默认聊天窗口增加“猜你想问”快捷问题区，帮助用户一键将常见问题模板填入输入框，再按实际场景补充 IP、端口、系统名称等参数。

## 目标

- 在默认聊天窗口首屏提供清晰的常见问题入口
- 让用户点击问题卡片后直接将模板填入输入框
- 保持正常聊天流程不变，不自动发送问题
- 仅在新会话首屏展示，避免干扰已有会话
- 使用前端配置驱动，便于后续按分类扩展问题模板

## 非目标

- 本次不接入后端接口维护快捷问题配置
- 本次不做个性化推荐、热门排序或搜索
- 本次不做自动发送
- 本次不改动现有聊天流式响应协议

## 用户场景

### 场景一：故障诊断

用户首次打开默认聊天页，希望快速发起“10.0.1.10 到 10.0.2.20 的 80 端口不通”这类诊断问题。用户点击对应卡片后，模板进入输入框，再手动修改具体参数并发送。

### 场景二：访问关系查询

用户需要查询某个系统的访问关系清单，但不知道该如何组织提问。用户点击“查询系统访问关系清单”后，输入框中出现标准提问模板。

### 场景三：权限提单

用户需要了解访问关系开通流程、提单权限和必填项。用户点击对应卡片后，输入框出现完整问题模板，并可继续补充系统名称或上下文。

## 方案概述

在默认聊天页增加一个前端渲染的“猜你想问”区块：

- 展示位置：欢迎消息下方、输入框上方
- 展示时机：无历史会话的新会话首屏
- 展示形式：按分类展示快捷问题卡片
- 卡片内容：标题 + 一行说明
- 点击结果：完整问题模板填入输入框，不自动发送
- 生命周期：发送首条消息后隐藏；点击“新对话”后重新出现

## 信息架构

快捷问题区包含以下层级：

1. 区块标题：`猜你想问`
2. 区块说明：`点击即可填入输入框，可继续补充具体 IP、端口、系统名称`
3. 分类组：
   - `故障诊断`
   - `访问关系`
   - `权限提单`
4. 分类下的快捷问题卡片

## 配置模型

前端采用配置驱动，配置结构定义在 `static/app.js` 中。推荐数据结构如下：

```js
const QUICK_PROMPTS = [
  {
    category: '故障诊断',
    items: [
      {
        title: '源到目标端口不通',
        description: '排查网络路径、防火墙、ACL 和服务监听',
        template: '请帮我分析为什么 10.0.1.10 到 10.0.2.20 的 80 端口不通。'
      }
    ]
  }
];
```

字段说明：

- `category`：分类名称
- `items`：分类下卡片集合
- `title`：卡片主标题
- `description`：卡片一行说明
- `template`：点击后写入输入框的完整问题模板

## 推荐首版内容

### 故障诊断

- 标题：`源到目标端口不通`
  说明：`排查网络路径、防火墙、ACL 和服务监听`
  模板：`请帮我分析为什么 10.0.1.10 到 10.0.2.20 的 80 端口不通。`
- 标题：`主机间网络不通`
  说明：`检查连通性、路由和安全策略`
  模板：`请帮我分析为什么 10.0.1.10 无法访问 10.0.2.20。`
- 标题：`服务监听异常`
  说明：`确认端口监听、进程状态和主机防火墙`
  模板：`请帮我检查 XX 主机上的 XX 服务是否正常监听目标端口。`

### 访问关系

- 标题：`查询系统访问关系清单`
  说明：`查看某系统的上下游访问关系`
  模板：`请查询 XX 系统的访问关系清单，包括源系统、目标系统、协议、端口和访问方向。`
- 标题：`查询主机访问关系`
  说明：`按 IP 查看与其他资产的访问关系`
  模板：`请查询 IP 为 10.0.1.10 的主机有哪些访问关系。`
- 标题：`查询两个系统是否已开通`
  说明：`确认系统间是否已有访问放通记录`
  模板：`请帮我查询 XX 系统到 XX 系统之间是否已经开通访问关系。`

### 权限提单

- 标题：`访问关系如何开通提单`
  说明：`查看权限、流程和必填信息`
  模板：`访问关系如何进行开通提单？需要哪些权限、审批节点和必填信息？`
- 标题：`提单需要准备什么`
  说明：`提前准备源目地址、端口和用途说明`
  模板：`开通访问关系前需要准备哪些信息，例如源 IP、目标 IP、端口、协议和用途说明？`
- 标题：`谁有权限提单`
  说明：`确认申请角色和审批边界`
  模板：`哪些角色有权限发起访问关系开通提单？审批边界是什么？`

## 组件设计

### HTML

在默认聊天页新增快捷问题容器，推荐以独立区块插入欢迎消息区域之后。容器包含：

- 标题区域
- 说明文本
- 分类列表
- 卡片列表

不需要增加新的页面路由或额外页面。

### CSS

视觉风格延续现有聊天页风格，但弱于主输入区。要求：

- 浅色底、轻边框、圆角卡片
- hover 时边框高亮并轻微上浮
- 桌面端一行 2 到 3 个卡片
- 移动端单列
- 文案最多显示两行，避免首屏过高

### JavaScript

推荐新增以下前端方法：

- `renderInitialState()`：渲染欢迎消息和快捷问题区
- `renderQuickPrompts()`：按配置渲染分类和卡片
- `hideQuickPrompts()`：发送首条消息后隐藏
- `showQuickPrompts()`：新对话时恢复展示
- `fillPromptTemplate(template)`：将模板写入输入框并触发高度重算
- `hasConversationHistory()`：判断当前是否存在历史消息或已恢复会话

## 交互流程

### 首次进入页面

1. 页面加载
2. 检查 URL 中是否存在 `session_id`
3. 检查本地是否存在可恢复会话
4. 若无历史会话，则渲染欢迎消息和快捷问题区
5. 若有历史会话，则按现有逻辑恢复消息，不渲染快捷问题区

### 点击快捷问题

1. 用户点击卡片
2. 将该卡片的 `template` 写入输入框
3. 调用输入框自动高度调整逻辑
4. 将焦点定位到输入框
5. 不触发表单提交

### 发送首条消息

1. 用户编辑输入框内容
2. 提交消息前隐藏快捷问题区
3. 进入现有聊天提交流程
4. 后续会话中不再展示快捷问题区

### 点击新对话

1. 清理当前会话状态和本地缓存
2. 清空消息列表
3. 重新渲染欢迎消息
4. 重新渲染快捷问题区
5. 输入框恢复可编辑状态

## 状态规则

应展示快捷问题区的条件：

- 当前没有 `session_id`
- 本地没有可恢复消息
- 当前消息列表为空或仅包含首屏欢迎内容

不应展示快捷问题区的条件：

- 正在从服务端恢复历史会话
- 本地已恢复聊天记录
- 用户已发送至少一条消息

## 错误处理

- 若快捷问题配置为空，则不渲染该区块，不影响正常聊天
- 若单张卡片模板缺失，则该卡片不展示
- 若用户点击卡片时输入框不存在，则安全退出并打印前端日志
- 若恢复历史会话失败，仍按现有错误处理逻辑执行，避免误展示与实际会话冲突的首屏引导

## 编码与兼容性注意事项

当前静态页面存在中文乱码迹象。实施时应统一确认以下文件使用 UTF-8 编码保存：

- `static/index.html`
- `static/app.js`
- `static/style.css`

若不统一编码，新增中文文案可能继续出现乱码。

## 测试方案

### 手工验证

- 首次打开默认聊天页，展示欢迎消息和快捷问题区
- 点击任一卡片后，模板进入输入框但不自动发送
- 用户可继续修改模板并提交
- 发送第一条消息后，快捷问题区隐藏
- 刷新并恢复已有会话时，不展示快捷问题区
- 点击“新对话”后，欢迎消息和快捷问题区重新出现
- 移动端宽度下卡片单列展示，输入框仍可见

### 回归点

- 不影响现有流式聊天
- 不影响停止生成按钮
- 不影响本地会话恢复
- 不影响通过 `session_id` 加载历史会话
- 不影响现有欢迎消息和系统消息展示

## 实施范围

本次实现仅涉及以下文件：

- `static/index.html`
- `static/style.css`
- `static/app.js`

不涉及后端接口、数据库结构和会话协议调整。

## 后续演进

后续如需扩展，可按以下方向演进：

- 将快捷问题配置从前端迁移至后端接口
- 按用户角色、系统域或最近使用频率返回不同问题
- 增加“更多问题”或搜索能力
- 支持后台维护快捷问题模板

## 决策结论

本设计采用前端配置驱动方案，在默认聊天窗口首屏提供分类化快捷问题入口。该方案实现成本低、对现有架构侵入小、可快速验证用户使用效果，并为后续升级为后端动态配置保留清晰演进路径。
