| **总计** | **平均 1-3 小时** |

> 大量时间花在"查"和"试"上

# 03 技术架构

---

## 整体架构

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│   Web界面   │  │  REST API   │  │     CLI     │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       └────────────────┴────────────────┘
                          │
                          ▼
       ┌──────────────────────────────────┐
       │        双引擎业务层               │
       │  ┌──────────┐    ┌──────────┐   │
       │  │规则引擎  │ or │LLM Agent │   │
       │  │(80%)     │    │(20%)     │   │
       │  └──────────┘    └──────────┘   │
       └──────────────────────────────────┘
                          │
       ┌──────────────────────────────────┐
       │      RAG知识库（本地）            │
       │  BGE嵌入 + ChromaDB向量存储       │
       └──────────────────────────────────┘
                          │
       ┌──────────────────────────────────┐
       │         执行层                    │
       │   CMDB查询  │  命令执行  │  网络设备  │
       └──────────────────────────────────┘
```

---

## 关键技术决策

| 决策 | 方案 | 理由 |
|-----|------|------|
| **嵌入模型** | BGE-small-zh (本地) | 中文优化，512维，完全本地 |
| **向量数据库** | ChromaDB | 轻量，无需额外服务 |
| **LLM框架** | LangChain | 生态成熟，工具调用方便 |
| **会话存储** | SQLite | 简单可靠，零配置 |

---

## 数据安全设计

<div class="metric">

✅ **故障描述** — 支持脱敏（IP替换为标识符）
✅ **RAG检索** — 完全本地，数据不出内网

✅ **命令执行** — 白名单机制，只读/安全命令
✅ **审计日志** — 全链路可追溯

# 04 演示

---

## 场景1：端口不通

### 规则引擎秒级响应

**输入：** `10.0.1.10到10.0.2.20端口80不通`

**过程：**
1. NLU解析 → 提取IP、端口
2. 规则匹配 → "端口连通性"模式，置信度0.95
3. 直接输出 → **目标主机端口未监听**

**耗时：< 3秒**
**API成本：0**

---

## 场景2：复杂跨域问题

### LLM Agent + RAG

**输入：** `上海机房访问北京机房数据库延迟高，偶尔超时`

**过程：**
1. 规则引擎置信度0.45 → 转入Agent模式
2. Task Planner生成排查计划
3. 调用CMDB获取网络拓扑
4. 执行 traceroute 定位延迟节点
5. **RAG检索** → 匹配历史案例"跨省专线抖动处理"

---

## 场景3：知识库管理

**操作：** 拖拽上传运维文档

支持的文档类型：
- 防火墙配置规范 (.txt)
- 故障处理SOP (.txt)
- 历史案例分析 (.txt)

**自动处理：**
文档 → 智能分割 → 向量化 → 存入ChromaDB

# 05 落地计划

---

## 预期价值

| 指标 | 现状 | 目标 |
|-----|------|------|
| **常见故障MTTR** | 平均1-3小时 | **降至10分钟** |
| **新人上手周期** | 3-6个月 | **缩短至1个月** |
| **夜间值班压力** | 高频人工排查 | **70%场景自动定位** |

---

## 三阶段落地

### Phase 1（1个月）
- 核心引擎就绪
- 选择1个网络区域试点
- 收集反馈、优化规则库

### Phase 2（2个月）
- 接入内部LLM
- 推广到全部网络区域
- 完善知识库内容

### Phase 3（3个月）
- 对接告警系统（自动触发诊断）
- 对接工单系统（结果自动填单）
- 形成闭环

---

## 资源需求

<div class="metric">

**硬件**
- 1台GPU服务器（内部LLM部署）或 继续使用现有API

**人力**
- 2个后端开发（引擎完善、内部系统对接）
- 1个运维专家（知识库建设、规则补充）

</div>

---

## 风险与对策

| 风险 | 对策 |
|-----|------|
| LLM幻觉给出错误建议 | 规则引擎兜底；低置信度转人工；人工确认后执行 |
| 敏感数据泄露 | 脱敏处理；RAG完全本地；LLM优先内部部署 |
| 工程师抵触新工具 | 定位为"助手"非"替代"；保留人工决策权；快速迭代 |
